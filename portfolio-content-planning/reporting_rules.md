# Reporting Rules: Portfolio Content Ledger

## Purpose
このチャットでは、ポートフォリオWebのデザイン・コーディングには進まず、台帳、掲載内容、文章、素材、検討事項の整理だけを行う。

今後の応答では、作業内容だけでなく、必ずタスク状態と残りの判断事項を併記する。

## Mandatory Response Format
作業報告をする場合は、原則として次の順に書く。

1. **結論**
   - 今回どこまで進んだか。
   - Web実装に進んでいないこと。
2. **タスク状況**
   - 完了、今回更新、継続、未完了を分ける。
   - `portfolio_content_review.xlsx` の `00_進捗と残タスク` と対応させる。
3. **今回更新した内容**
   - 追加・修正したシート、文言、検討事項、素材整理を短く列挙する。
4. **未確定・ユーザー判断**
   - 公開範囲、Project統合、リンク配置、最終語調など、判断が必要なものを明示する。
5. **確認先**
   - 最終成果物のExcelへのリンクを出す。

## Must Not Do
- このチャットで `src` へのWeb反映、デザイン調整、コーディングを始めない。
- 「大丈夫です」だけで終えない。
- タスク表を消した報告をしない。
- 検討事項をチャットだけで流さず、MarkdownまたはExcel台帳に残す。
- ユーザーが全シートを精読する前提にしない。判断用シートと実装参照用シートを分けて説明する。

## Current Naming Decisions
- `PROJECTS INDEX` は章名として使わない。章名は `PRACTICE ARCHIVE` とする。
- `PRACTICE ARCHIVE` は、仕事・研究・教育・地域連携を束ねた実績群を指す。
- `Index` はナビゲーションやメニュー機能名としてのみ使う。
- 右上の分数表示はトップページ主要章だけに使う。下層ページでは分数を使わず、カテゴリ名だけを表示する。
- 台帳ではProject / Year / Category / Role / Client / Description / Proof / Assets / Links等を共通保持するが、トップページでは表示項目を絞る。
- Selected ProjectsとPractice Archiveは同じ情報モデルから出す。違いは表示密度と見せ方。
- トップページの表示文言は最小限にする。証拠、公開注意、リンク、長い説明は下層詳細へ送る。
- Archiveは台帳上では全件に詳細原稿を用意する。実装時には、一覧に掲載するか、クリック可能な詳細ページを設けるかを別々に判断する。
- 見出しは編集的でよいが、リードと本文は対象、制作物、役割、用途が見える具体文にする。
- `てがかり`、`ひらく`、`つなぐ`、`支える`、`未来`、`可能性`、`問い` は、具体的な対象や制作行為が同じ文脈にない場合は避ける。

## Current Important Pending Decisions
- Selected Projectsは5件固定。6件目は補充しない。
- 公開範囲はユーザー確認済み。Web化時に必要な最終トリミング・マスクだけ確認する。
- Junior Law School Okayamaの制作年は2024で確定。主催・開催情報を詳述する場合だけ追加確認する。
- JSET詳細にはJ-STAGE当該ページ、JSET研究会発表ページ、Web用PDFとして配置する使用スライドをリンクしてよい。
- JSET詳細の主画像セットは、ZINE主画像 + 表紙 + 見開き2-3点 + JSET資料で進める。制作環境画像は必要な場合のみ補助素材として扱う。
- Mori no Geijutsusaiは、学生活動写真がない場合も鑑賞ガイドとFD資料中心で構成する。
- 最終的な言語トーン。
- Practice Archive内の各実績について、一覧掲載するか、詳細ページ化するか。
